### SVI on-chain pricing formula (oracle.move compute_nd2)
k = ln(strike/forward) (predict_math::ln, 1e9 fixed-point); w(k) = a + b*(rho*(k−m) + sqrt((k−m)² + sigma²)) — all SVI params 1e9-scaled, rho and m are deepbook_predict::i64::I64 signed structs {magnitude u64, is_negative bool}; d2 = −((k + w/2)/sqrt(w)); price = predict_math::normal_cdf(d2). Aborts: EZeroForward(3) if forward==0, ECannotBeNegative(4) if rho*(k−m)+sqrt(...)<0, EZeroVariance(5) if w==0. ### Settled-oracle accounting subtlety for redeem ordering
redeem/redeem_range branch on `oracle.is_settled() && vault.has_settled_oracle(oracle.id())` — i.e. settled-path accounting only applies after predict::compact_settled_oracle has moved the oracle into vault.settled_oracles (SettledOracleState{remaining_quantity, remaining_liability}). If the oracle is settled on-chain but NOT yet compacted, redeem takes the live path: remove_position + refresh_oracle_risk, where refresh_oracle_risk itself calls set_mtm_with_settlement for settled oracles — payout is still settlement-correct (compute_price returns 0/1e9), but quoting asserts assert_quoteable_oracle (oracle settled OR fresh) so settled positions remain redeemable despite staleness. For the range ladder: redeem_range after expiry pays exactly 1e9·qty in-band / 0 out-of-band with ask==bid==fair (zero spread).

## Gotchas
- The repo-shipped CONFIG ids in predict_workshop/*.ts (MANAGER_ID 0x51f0..., ORACLE_ID 0x57ab..., expiries 2026-05-28/29) are workshop leftovers — managers are per-address and those oracles have expired/compacted; always create your own manager and discover oracles via the indexer (status==='active' && expiry>now && settlement_price==null).
- create_manager makes the PredictManager a SHARED object, but mint/redeem/mint_range/redeem_range/deposit/withdraw assert ctx.sender()==owner (ENotOwner=1); only redeem_permissionless (directional, settled-only) is callable by third parties — there is NO permissionless variant for ranges.
- Two different scales in every call: strikes/prices/SVI = 1e9, quantities/DUSDC/PLP = 1e6. Mixing them is the #1 failure mode; the workshop scripts take human units and scale internally.
- mint debits the PredictManager's internal balance, not a Coin argument — you must predict_manager::deposit first (or SKIP_TOPUP with existing balance). Cost is quoted POST-trade (your own mint widens the price you pay), so a big ladder mint in one PTB pays increasing marginal asks per rung; quantity-aware previews need get_trade_amounts at the intended size, and ask must stay within ask_bounds (default 1%–99%) or the whole mint aborts EAskPriceOutOfBounds(7).
- Oracle quotes go stale 30_000 ms after the last keeper update (EOracleStale=6 in assert_live/quoteable_oracle): mints/redeems revert if the oracle-feed lapses; the strategy executor needs retry-on-stale logic.
- Strikes must lie exactly on the oracle grid: snapped to tick_size from min_strike, within min_strike + 100_000 ticks (EInvalidStrike=2). Read min_strike/tick_size from /predicts/{id}/oracles or the OracleSVI object before building keys; ranges additionally need lower<higher (EInvalidStrikes=0).
- transactions/predict/deployOracles.ts imports '../../config/predict-oracles.js' but scripts/config/ contains only constants.ts in BOTH clones — the import target is generated by the script itself on success, so the first run fails at import until you stub the file (export const predictOracles = {testnet: [], mainnet: []}).
- dUSDC has no faucet: minting requires the TreasuryCap 0x64f8a47a0af0a3b14db3a7ce89aa206ff77a9c6b5ac0eaef6db2ea46da3ced94 owned by the workshop host; if your key doesn't own it, dusdc-mint fails and you need a transfer from the host.
- @mysten/sui ^2.5.0 API differs from older sdks: client class is SuiJsonRpcClient from '@mysten/sui/jsonRpc' (not SuiClient/getFullnodeUrl from '/client'), decodeSuiPrivateKey may return undefined schema (default to Ed25519), and sdk default tx expiration ValidDuring breaks older tools (set {Epoch: n+5} if serializing).
- No script ever sets a gas budget — SDK auto-estimation is the convention on testnet; only mainnet multisig flows need GAS_OBJECT + prepareMultisigTx.
- Workshop listPositions reconstructs open RANGE positions client-side from /ranges/minted minus /ranges/redeemed (the manager summary endpoints only cover directional positions) — replicate that net-by-(oracle_id|expiry|lower|higher) logic in the terminal.
- registry::create_oracle does NOT add the passed cap to the oracle's authorized_caps (boots empty); register_oracle_cap (AdminCap-gated) must run before activate/update_svi/update_prices — relevant if the team runs its own oracle for demos.
- predict::redeem on a settled-but-not-yet-compacted oracle takes the live-path vault accounting (has_settled_oracle false until compact_settled_oracle runs); payout values are still settlement-correct but vault MTM/risk numbers differ until the keeper compacts — Monte-Carlo dashboard should treat settled-uncompacted oracles separately.
- vault_value() ABORTS (EMtmExceedsBalance) if total_mtm > balance; a devInspect of supply/withdraw previews can revert in extreme-utilization states — guard the dashboard's quote calls.
- PLP withdrawal capacity = balance − total_max_payout (not vault_value), plus a rate limiter that admins may enable; show available_withdrawal(predict, clock) rather than naive share-value for 'can I exit now' UX.
- SVI params pushed on-chain need ≥9-decimal precision upstream; the indexer's latest_svi may show tiny a values for short-dated expiries — recompute prices in 1e9 ints exactly like compute_nd2 (ln, sqrt, normal_cdf from helper/math.move) or surface/no-arb checks will show phantom violations vs on-chain asks.
- Settlement trigger is implicit: the FIRST oracle::update_prices after expiry freezes settlement_price (the keeper's 'settle nudge'); there is no public settle() entrypoint, and a 5s safety window before expiry drops oracles from price batches.
- scripts/utils/utils.ts and scripts/config/constants.ts are byte-identical between deepbookv3-predict and deepbookv3-workshop clones; the workshop repo = predict repo + transactions/predict_workshop/ + extra pnpm aliases, so either clone works as the dependency base.
