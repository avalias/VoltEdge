## Gotchas
- No public dUSDC faucet: TreasuryCap 0x64f8...ced94 is AddressOwner 0xcca26f7ae2e40604498294e95bacccc4652cc8cb2aa074d7ee608c7e7bdf0c29 (live RPC-verified); every repo script assumes the ops PRIVATE_KEY. The range-ladder strategy needs a dUSDC sourcing plan before any mint/redeem PTB can run.
- No TS SDK for predict exists anywhere (npm or repo) — @mysten/deepbook-v3@1.2.1 covers only the CLOB. Plan to generate bindings with @mysten/codegen (the README-blessed path) or copy raw moveCall targets from scripts/transactions/predict/*.ts and packages/predict/simulations/src/runtime.ts.
- Testnet deployment is rotated by redeploy.ts ('pnpm predict-redeploy' rewrites constants.ts; README explicitly says ignore old package IDs). Hardcoded package/object IDs can die mid-hackathon — fetch /predicts/:id/state and /config from predict-server at startup and make IDs configurable.
- Local git history is one squashed commit (2026-04-27 snapshot of branch predict-testnet-4-16); upstream MystenLabs/deepbookv3 activity and drift must be checked on GitHub, not locally. The vendored branch is AHEAD of the upstream main migration plan (PREDICT_MIGRATION.md shows only PR1-2 merged to main at doc time).
- Mixed scaling: prices/strikes/SVI params/spreads are 1e9 fixed-point (float_scaling), but quantities/payouts/PLP are 6-decimal USDC units (1_000_000 = 1 contract = $1). Strikes on the wire are e.g. 50_000 * 1e9. Off-by-1e3 errors between these two regimes are the most likely terminal bug.
- Oracle staleness threshold is 30s on-chain and the feed pushes prices every ~1s but SVI much less often; mints near expiry can also fail because the feed stops pushing 5s (SAFETY_WINDOW_MS) before expiry and the first post-expiry push settles the oracle. The ladder bot must handle aborts from stale/settled oracles.
- MarketKey identity check: market_key::assert_matches_oracle takes &OracleSVI (non-generic; oracle carries underlying_asset: String) — the generic OracleSVI<Underlying> form from the old at/predict branch is dead; don't copy from stale upstream PRs.
- predict-server is GET-only and 'low-lag, not zero-lag' — after PTB execution, confirm via on-chain object reads, then refresh indexer endpoints; don't assert indexer state in the same tick as the tx.
- 'iron_bank' appears nowhere in this repo — if the hackathon brief mentions it, it is external to this codebase; the only margin-adjacent composition substrate is the shared deepbook::balance_manager that both PredictManager and deepbook_margin build on.
